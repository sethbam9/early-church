# AI Data Edit Workflow

Use this process whenever adding, updating, or removing records across the data TSV files.

## Source files (edit directly)

| File | What it contains |
|------|-----------------|
| `data/people.tsv` | Historical figures |
| `data/polities.tsv` | Political entities |
| `data/persuasions.tsv` | Theological traditions/streams |
| `data/cities.tsv` | Canonical city records + static church-planting metadata |
| `data/works.tsv` | Primary source texts |
| `data/events.tsv` | Historical events |
| `data/doctrines.tsv` | Theological doctrines |
| `data/quotes.tsv` | Verbatim evidence quotes |
| `data/archaeology.tsv` | Physical sites/artefacts |
| `data/relations.tsv` | Graph edges between entities |
| `data/notes.tsv` | Evidence notes (content-hash ID) |
| `data/place_state_by_decade.tsv` | Temporal map state per place/decade |

## Derived files (NEVER edit manually — regenerated by `npm run data:validate`)

| File | Script | Source |
|------|--------|--------|
| `data/places.tsv` | `scripts/derive_places.ts` | `cities.tsv` + `archaeology.tsv` |
| `data/entity_place_footprints.tsv` | `scripts/derive_footprints.ts` | relations, works, events, archaeology, quotes, place_state_by_decade |
| `data/note_mentions.tsv` | `scripts/derive_mentions.ts` | `notes.tsv` body_md `[[type:id]]` tags |

## Rules

1. **IDs are stable slugs** — never change a `*_id` once it is referenced by other files.
2. **FK consistency** — if you rename an ID, find-and-replace in ALL files that reference it.
3. **No literal `null`** — use empty string for missing values.
4. **Semicolons** separate multi-value fields (e.g. `roles`, `persuasion_ids`, `name_alt`).
5. **`notes.tsv` note_id** is a 10-char SHA-1 content hash — do not invent these; the validator checks referential integrity but not the hash itself.
6. **Derived files** — never edit `places.tsv`, `entity_place_footprints.tsv`, or `note_mentions.tsv` manually. They are always regenerated by `npm run data:validate`.
7. **Check `docs/enum-schema.md`** before using any enum value. If you need a new value, add it to the schema doc first, then use it in the data file.
8. **No semantic duplicates** — before adding any record, search for existing records that express the same meaning. See the "Semantic Dedup Checklist" section below.

## Doctrine stance — the key input for the doctrine city map

Whenever you add a relation that links a person or work to a doctrine, **always set `polarity`**:
- `supports` — the person/work affirms this doctrine
- `opposes` — the person/work condemns/rejects this doctrine
- `neutral` — factual association without clear stance

This single field drives the automatic derivation of all doctrine→city footprints. You do NOT need to manually add any rows to `entity_place_footprints.tsv`.

**Example**: Adding `person:tertullian → condemns → doctrine:infant-baptism (polarity=opposes)` automatically generates footprints for every city Tertullian was `bishop_of` or `active_in`, with `stance=condemns`.

## Commit-sized batch process

1. **Identify the change set** — e.g. "add 3 new people", "populate persuasion descriptions", "rename polity ID".
   1.1 Before adding a new record, check for similar slugs first to avoid duplicates.
2. **Edit the TSV files** using the Edit or multi_edit tools. Keep each batch ≤ 20 rows to stay reviewable.
3. **Validate** — this auto-derives all derived files and then checks FKs / headers:
   ```
   npm run data:validate
   ```
4. **Fix any errors** reported before moving to the next batch.
5. Repeat steps 2–4 until the full change is done.

## ID renaming checklist

When renaming a primary key (e.g. `polity_id`, `person_id`):
- [ ] Update the ID in its own TSV file
- [ ] `data/place_state_by_decade.tsv` — `polity_id` column
- [ ] `data/notes.tsv` — `[[type:id]]` mention tags in `body_md`
- [ ] `data/relations.tsv` — `source_id` / `target_id` columns
- [ ] `data/events.tsv` — `key_figure_person_ids` column (for persons)
- [ ] `data/works.tsv` — `author_person_id` column (for persons)
- [ ] `data/people.tsv` — `city_of_origin_id` column (for cities)
- [ ] `data/archaeology.tsv` — `city_id` column (for cities)
- [ ] `data/quotes.tsv` — `doctrine_id` / `work_id` columns
- [ ] `data/essays/*.md` — `[[type:id]]` mention tags
- [ ] **Do NOT update** `entity_place_footprints.tsv`, `note_mentions.tsv`, `places.tsv` — derived files are regenerated automatically

## Adding a new city

1. Add row to `data/cities.tsv` — include `church_planted_by` and `apostolic_origin_thread` if known
2. `places.tsv` will be auto-derived — **do not add a row manually**
3. Optionally add rows to `data/place_state_by_decade.tsv`
4. Run validator (`npm run data:validate`)

## Adding a new archaeology site

1. Add row to `data/archaeology.tsv`
2. `places.tsv` will be auto-derived — **do not add a row manually**
3. Run validator (`npm run data:validate`)

## Adding a new relation_type

Whenever you introduce a **new `relation_type` slug** in `data/relations.tsv`, you **must** add a matching entry to `src/domain/relationLabels.ts`:

```ts
// in RELATION_LABELS:
my_new_type: { forward: "human forward label", inverse: "human inverse label" },
```

After updating the file run the TypeScript build to catch any type errors:
```
npm run build
```

## Adding a new person

1. Add row to `data/people.tsv`
2. If they wrote works, add rows to `data/works.tsv` with `author_person_id`
3. If they appear in events, update `key_figure_person_ids` in `data/events.tsv`
4. **Add doctrine stance relations** to `data/relations.tsv` if they affirm/condemn doctrines (this is the key input for the doctrine city map)
5. Add city relations (`bishop_of`, `active_in`, etc.) to `data/relations.tsv`
6. Run validator

## Adding a new note

Notes use `[[type:id|label]]` mention tags in `body_md`. When you save the note and run validate:
- The validator checks that all `[[type:id]]` references resolve to real entities
- `note_mentions.tsv` is automatically re-derived (no manual editing needed)

## Semantic Dedup Checklist

Before adding **any** new record, run these checks:

### People
- `grep -i "<surname or key term>" data/people.tsv` — check for variant spellings, alternate IDs

### Relations
- **Directional relations are one-way only.** Do NOT add both `A teacher_of B` and `B disciple_of A`. Pick the canonical direction per `docs/enum-schema.md`.
- **Symmetric relations (`co-worker`, `corresponded_with`)**: add only ONE row. Use alphabetical source ID as the tiebreaker.
- **`bishop_of`**: only the person→city direction.
- Before adding a relation, search: `grep "<source_id>.*<target_id>\|<target_id>.*<source_id>" data/relations.tsv`

### Works
- Check `grep -i "<author or title keyword>" data/works.tsv` for existing entries

### Events
- Search by year range and keyword: `grep "<keyword>" data/events.tsv`

### Enum values
- **Always check `docs/enum-schema.md`** before using a value
- If introducing a new value, add it to the schema doc **first**
