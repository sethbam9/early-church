# AI Data Edit Workflow

Use this process whenever adding, updating, or removing records across the data TSV files.

## Source files (edit directly)

| File | What it contains |
|------|-----------------|
| `data/people.tsv` | Historical figures |
| `data/polities.tsv` | Political entities |
| `data/persuasions.tsv` | Theological traditions/streams |
| `data/cities.tsv` | Canonical city records |
| `data/places.tsv` | **DERIVED** ‚Äî auto-generated from cities + archaeology (do NOT edit) |
| `data/works.tsv` | Primary source texts |
| `data/events.tsv` | Historical events |
| `data/doctrines.tsv` | Theological doctrines |
| `data/quotes.tsv` | Verbatim evidence quotes |
| `data/archaeology.tsv` | Physical sites/artefacts |
| `data/relations.tsv` | Graph edges between entities |
| `data/notes.tsv` | Evidence notes (content-hash ID) |
| `data/note_mentions.tsv` | Pre-parsed note‚Üíentity joins |
| `data/place_state_by_decade.tsv` | Temporal map state per place/decade |
| `data/entity_place_footprints.tsv` | Entity‚Üîplace associations |

## Rules

1. **IDs are stable slugs** ‚Äî never change a `*_id` once it is referenced by other files.
2. **FK consistency** ‚Äî if you rename an ID, find-and-replace in ALL files that reference it.
3. **No literal `null`** ‚Äî use empty string for missing values.
4. **Semicolons** separate multi-value fields (e.g. `roles`, `persuasion_ids`, `name_alt`).
5. **`notes.tsv` note_id** is a 10-char SHA-1 content hash ‚Äî do not invent these; the validator checks referential integrity but not the hash itself.
6. **`places.tsv` is derived** ‚Äî it is auto-generated by `npm run data:derive-places` from `cities.tsv` + `archaeology.tsv`. **Never edit it manually.** The derive script runs automatically as part of `npm run data:validate`.
7. **Check `docs/enum-schema.md`** before using any enum value. If you need a new value, add it to the schema doc first, then use it in the data file.
8. **No semantic duplicates** ‚Äî before adding any record, search for existing records that express the same meaning. See the "Semantic Dedup Checklist" section below.

## Commit-sized batch process

1. **Identify the change set** ‚Äî e.g. "add 3 new people", "populate persuasion descriptions", "rename polity ID".
1.1 Before adding a new record, check the records with a similar slug first to make sure you are not duplicating. For example don't add "paul" if there is already "paul-apostle".
2. **Edit the TSV files** using the Edit or multi_edit tools. Keep each batch ‚â§ 20 rows to stay reviewable.
3. **Validate** ‚Äî this auto-derives `places.tsv` and then checks FKs / headers:
   ```
   npm run data:validate
   ```
4. **Fix any errors** reported before moving to the next batch.
5. Repeat steps 2‚Äì4 until the full change is done.

## ID renaming checklist

When renaming a primary key (e.g. `polity_id`, `person_id`):
- [ ] Update the ID in its own TSV file
- [ ] `data/place_state_by_decade.tsv` ‚Äî `polity_id` column
- [ ] `data/note_mentions.tsv` ‚Äî `mentioned_slug` column
- [ ] `data/notes.tsv` ‚Äî `[[type:id]]` mention tags in `body_md`
- [ ] `data/entity_place_footprints.tsv` ‚Äî `entity_id` column
- [ ] `data/relations.tsv` ‚Äî `source_id` / `target_id` columns
- [ ] `data/events.tsv` ‚Äî `key_figure_person_ids` column (for persons)
- [ ] `data/works.tsv` ‚Äî `author_person_id` column (for persons)
- [ ] `data/people.tsv` ‚Äî `city_of_origin_id` column (for cities)
- [ ] `data/archaeology.tsv` ‚Äî `city_id` column (for cities)

## Adding a new city

1. Add row to `data/cities.tsv`
2. `places.tsv` will be auto-derived ‚Äî **do not add a row manually**
3. Optionally add rows to `data/place_state_by_decade.tsv` and `data/entity_place_footprints.tsv`
4. Run validator (`npm run data:validate`)

## Adding a new archaeology site

1. Add row to `data/archaeology.tsv`
2. `places.tsv` will be auto-derived ‚Äî **do not add a row manually**
3. Run validator (`npm run data:validate`)

## Adding a new relation_type

Whenever you introduce a **new `relation_type` slug** in `data/relations.tsv`, you **must** add a matching entry to `src/domain/relationLabels.ts`:

```ts
// in RELATION_LABELS:
my_new_type: { forward: "human forward label", inverse: "human inverse label" },
```

- `forward` ‚Äî shown when the current entity is the **source** of the relation
- `inverse` ‚Äî shown when the current entity is the **target** of the relation

If no entry exists for a slug the UI falls back to `slug.replace(/_/g, ' ')`, which is acceptable but not ideal.

**Examples of correct label pairs:**
| relation_type | forward | inverse |
|---|---|---|
| `disciple_of` | disciple of | teacher of |
| `teacher_of` | teacher of | student of |
| `bishop_of` | bishop of | had bishop |
| `participated_in` | participated in | included |
| `ordained` | ordained | ordained by |
| `affirms` | affirms | affirmed by |

After updating the file run the TypeScript build to catch any type errors:
```
npm run build
```

## Adding a new person

1. Add row to `data/people.tsv`
2. If they wrote works, add rows to `data/works.tsv` with `author_person_id`
3. If they appear in events, update `key_figure_person_ids` in `data/events.tsv`
4. Add relations to `data/relations.tsv` if needed
5. If a doctrine references this person's work as `first_attested_work_id`, add the work to `works.tsv` first
6. Run validator

## Semantic Dedup Checklist

Before adding **any** new record, run these checks:

### People
- `grep -i "<surname or key term>" data/people.tsv` ‚Äî check for variant spellings, alternate IDs
- Example: `felix` vs `felix-i` ‚Äî same Pope Felix I

### Relations
- **Directional relations are one-way only.** Do NOT add both `A teacher_of B` and `B disciple_of A`. Pick the canonical direction per `docs/enum-schema.md`.
- **Symmetric relations (`co-worker`, `corresponded_with`)**: add only ONE row. Use alphabetical source ID as the tiebreaker.
- **`bishop_of`**: only the person‚Üícity direction. Do NOT add city‚Üíperson.
- **`opposed`**: add only from the initiator's perspective. Do NOT add the mirror.
- Before adding a relation, search: `grep "<source_id>.*<target_id>\|<target_id>.*<source_id>" data/relations.tsv`

### Works
- Check `grep -i "<author or title keyword>" data/works.tsv` for existing entries
- Watch for short-ID vs long-ID duplicates (e.g. `acts-paul-thecla` vs `acts-of-paul-and-thecla`)

### Events
- Search by year range and keyword: `grep "<keyword>" data/events.tsv`
- Example: two different IDs for the destruction of the Temple in 70 AD

### Enum values
- **Always check `docs/enum-schema.md`** before using a value
- If introducing a new value, add it to the schema doc **first**
- Convention: `snake_case` for multi-word values, hyphens only within compound role tokens
